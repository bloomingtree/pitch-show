# 个人中心设计方案

## 1. 背景与目标

### 现状问题
- 当前 `/user/profile` 页面仅有「用户名 + 头像编辑 + 删除账户」，功能单一
- 用户无法查看自己的存储用量、月度配额、已分析歌曲数量
- 本地数据（IndexedDB）和云端数据（API）完全割裂，用户对数据分布无感知
- 分析完的歌曲散落在左侧列表，缺乏管理入口

### 设计目标
将个人中心升级为一个**数据看板 + 歌曲管理**的聚合页，让用户一目了然地掌握：
- 我是谁、我的会员状态
- 本地和云端各有多少数据、配额余量
- 我的歌曲库（区分来源，支持管理操作）

## 4. 各模块详细设计

### 4.1 用户信息卡

**展示内容：**
- 头像（圆形，点击可更换）
- 用户名（未设置时显示「未设置」）
- 邮箱
- 会员状态标签：
  - 免费用户：灰色标签「免费版」
  - Pro 用户：金色标签「Pro」，旁显到期日期

**交互：**
- 点击头像或「编辑」按钮 → 进入编辑模式（复用现有表单）
- 编辑模式下可修改用户名、头像 URL

### 4.2 数据概览

**数据来源：**
- 云端数据：`getStorageQuota()` → `{ storage_limit, storage_used, monthly_limit, monthly_used, is_premium }`
- 本地数据：`songDB.getAll()` → 获取 IndexedDB 中的歌曲总数

**展示内容：**

| 指标 | 数据来源 | 说明 |
|------|---------|------|
| 总歌曲 | `本地数 + 云端数 - 重复数` | 去重后的总歌曲数（首期可简化为本地数，不去做重） |
| 云端项目 | `storage_used` | 服务器上的项目数 |
| 本月已用 | `monthly_used / monthly_limit` | 本月分析次数/上限 |

**进度条：**
- 云端存储：`storage_used / storage_limit`，用进度条可视化
- 本月配额：`monthly_used / monthly_limit`，用进度条可视化
- Pro 用户配额显示为「无限制」，不展示进度条

**免费 vs Pro 对比提示（可选）：**
- 当配额接近上限时，在进度条下方显示一行文字：「升级 Pro 获取更多存储和无限分析次数」

### 4.3 我的歌曲

**设计考量 — 本地 vs 云端的区分：**

目前系统中本地和云端是两套独立的数据体系：
- **本地**：IndexedDB，key 为文件名，存储 File 对象 + 分析后的 notes
- **云端**：API 项目列表，每个项目有 ID、名称、分析状态、音频 URL

两者之间没有关联关系（没有唯一标识可以匹配同一首歌），因此首期设计如下：

**Tab 筛选：**
- 「全部」：显示本地歌曲列表（用户最常见的使用场景）
- 「本地」：仅显示 IndexedDB 中的歌曲
- 「云端」：仅显示 API 返回的项目列表

**列表项设计：**
```
┌──────────────────────────────────────────┐
│ 🎵  歌曲名称                             │
│     来源标签 [本地] 或 [云端]             │
│     分析日期 · 时长                       │
│                         [打开] [删除]    │
└──────────────────────────────────────────┘
```

- **本地歌曲**：显示文件名、分析日期，点击「打开」跳转到分析页并加载该歌曲
- **云端项目**：显示项目名、分析状态、创建日期，点击「打开」跳转并从 API 加载结果

**批量操作：**
- 云端项目支持批量删除（复用现有 `batchDeleteProjects` API）
- 本地歌曲支持逐个删除

### 4.4 账户设置

保留现有的：
- 退出登录按钮
- 删除账户按钮（带确认弹窗）

## 5. 数据获取策略

```
页面加载时并行请求：
  ├── profileApi.getProfile()         → 用户信息
  ├── getStorageQuota()               → 云端存储和月度配额
  ├── getProjectList({ limit: 50 })   → 云端项目列表
  └── songDB.getAll()                 → 本地歌曲列表
```

四个数据源互不依赖，可以并行加载，页面使用骨架屏或 loading 状态。

---

## 6. 本地与云端的关系说明

### 当前状态
- 本地（IndexedDB）和云端（API）**完全独立**，无同步机制
- Pro 分析会同时保存到两端，但无关联标识
- 用户在分析页左侧看到的列表来自 IndexedDB

### 首期方案（本 PR）
- 个人中心同时展示两端数据，用标签区分来源
- 不做同步、不做去重
- 用户可以在个人中心管理两端数据

### 后续演进方向
1. **关联标识**：Pro 分析完成后，将云端 project_id 写入 IndexedDB 记录，建立关联
2. **状态同步**：在个人中心提供「同步」按钮，将云端项目列表与本地对比
3. **统一列表**：分析页左侧的歌曲列表改为优先从云端加载，本地作为离线缓存

### 新增组件

| 组件 | 用途 |
|------|------|
| 无 | 所有 UI 在 ProfileView 内实现，保持简单 |

### 数据流

```js
// ProfileView.vue
const quota = ref(null)        // getStorageQuota()
const projects = ref([])       // getProjectList()
const localSongs = ref([])     // songDB.getAll()
const profile = ref({})        // profileApi.getProfile()
```

### 计算属性

```js
// 云端存储进度
const storagePercent = computed(() =>
  quota.value ? (quota.value.storage_used / quota.value.storage_limit * 100) : 0
)

// 本月配额进度
const monthlyPercent = computed(() =>
  quota.value ? (quota.value.monthly_used / quota.value.monthly_limit * 100) : 0
)

// 是否 Pro 用户
const isPremium = computed(() => quota.value?.is_premium ?? false)
```

## 8. 验证方式

1. 访问 `/user/profile`，确认用户信息卡正确展示头像、用户名、邮箱、会员状态
2. 数据概览卡片正确显示本地歌曲数、云端项目数、月度配额使用情况
3. 进度条准确反映 storage_used/storage_limit 和 monthly_used/monthly_limit
4. 歌曲列表 Tab 切换正常：「全部」「本地」「云端」各自过滤
5. 点击「打开」能正确跳转到分析页并加载对应歌曲
6. 删除操作正常：本地删除从 IndexedDB 移除，云端删除调用 API
7. 未登录时路由守卫正确拦截并跳转登录页
